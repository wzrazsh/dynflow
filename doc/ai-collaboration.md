# Dynamic Workflow Engine - AI 协作规则

## 核心原则

本文档定义了 AI 与开发者协作时必须遵守的规则。这些规则确保 AI 生成的代码符合项目标准，并与现有架构保持一致。

## 1. 代码生成规则

### 1.1 类型安全
- **必须** 使用 TypeScript，禁止使用 `any` 类型
- **必须** 为所有导出函数和类提供 JSDoc 注释
- **必须** 使用具体的类型，避免 `unknown` 或 `object`
- **必须** 使用 `.js` 扩展名进行导入（ESM 模块）

### 1.2 错误处理
- **必须** 处理所有异步操作的错误
- **必须** 提供清晰的错误信息，包含上下文
- **必须** 使用项目定义的错误类型（见 `src/errors.ts`）
- **禁止** 吞没错误或使用空的 catch 块

### 1.3 代码风格
- **必须** 遵循项目的 ESLint 配置
- **必须** 使用一致的命名约定：
  - 函数和变量：camelCase
  - 类和接口：PascalCase
  - 常量：UPPER_SNAKE_CASE
  - 文件名：kebab-case
- **必须** 使用下划线前缀标记未使用的参数（`_param`）

### 1.4 模块结构
- **必须** 在 `src/` 目录中使用 ESM 模块
- **必须** 在每个目录中创建 `index.ts` 进行桶导出
- **必须** 遵循单向依赖原则：
  ```
  types/ → llm/ → agent/ → runtime/ → builder/ → index.ts
  ```

## 2. 测试规则

### 2.1 测试生成
- **每次提供代码修改后，必须同时提供对应的单元测试代码**
- **必须** 使用项目的测试工具（见 `tests/helpers/mock-llm.ts`）
- **必须** 测试正常流程和错误流程
- **必须** 测试边界条件和边缘情况

### 2.2 测试命名
- **必须** 使用清晰的业务语义命名测试：
  ```typescript
  // ✅ 正确
  it('should lock account after 3 failed attempts', () => {...});
  
  // ❌ 错误
  it('test1', () => {...});
  it('should work', () => {...});
  ```

### 2.3 测试执行
- **在你认为改动可能影响其他模块时，主动列出需要重新运行的测试命令**
- **修复 Bug 时，先生成能复现 Bug 的测试用例，确保修复后该用例通过**
- **永远不要删除或跳过现有测试，除非明确指示**

### 2.4 测试覆盖
- **新代码必须达到 80% 的行覆盖率**
- **分支覆盖率必须达到 70%**
- **核心模块必须达到 90% 的覆盖率**

## 3. 文档规则

### 3.1 代码文档
- **必须** 为所有公共 API 提供 JSDoc 注释
- **必须** 包含参数说明、返回值说明和使用示例
- **必须** 标记废弃的 API 和替代方案

### 3.2 架构文档
- **必须** 更新相关架构文档（如果修改了架构）
- **必须** 记录重要的设计决策
- **必须** 保持文档与代码同步

### 3.3 示例代码
- **必须** 提供可运行的示例代码
- **必须** 包含注释说明关键步骤
- **必须** 展示最佳实践

## 4. 协作流程

### 4.1 任务开始前
1. **阅读相关文档**：`vision.md`、`design-doc.md`、`decisions.md`
2. **了解当前状态**：查看 `tasks.md` 中的待办事项
3. **确认测试策略**：查看 `testing/strategy.md`

### 4.2 任务执行中
1. **遵循代码规范**：严格遵守代码生成规则
2. **编写测试**：同步生成单元测试
3. **更新文档**：如果修改了 API 或架构，更新相关文档
4. **记录决策**：如果做出重要决策，更新 `decisions.md`

### 4.3 任务完成后
1. **类型检查**：运行 `npm run typecheck`
2. **构建**：运行 `npm run build`
3. **运行测试**：运行 `npm test`，确保所有测试通过
4. **代码规范**：运行 `npm run lint`
5. **更新任务状态**：更新 `tasks.md` 中的任务状态
6. **提供验证建议**：列出需要运行的测试命令

## 5. 禁止事项

### 5.1 代码禁止事项
- ❌ 使用 `any` 类型
- ❌ 使用 `@ts-ignore` 或 `@ts-expect-error`
- ❌ 吞没错误或使用空的 catch 块
- ❌ 在 `src/` 中使用 CommonJS 模块
- ❌ 硬编码配置值

### 5.2 测试禁止事项
- ❌ 删除或跳过现有测试
- ❌ 使用真实外部服务进行测试
- ❌ 编写无意义的测试（如 `it('should work', () => {...})`）
- ❌ 忽略测试失败

### 5.3 文档禁止事项
- ❌ 不更新过时的文档
- ❌ 编写无法运行的示例代码
- ❌ 忽略架构决策记录
- ❌ 不提供使用示例

## 7. 工具使用

### 7.1 必须使用的工具
- **TypeScript**：类型检查和编译
- **ESLint**：代码风格检查
- **Vitest**：单元测试和集成测试
- **tsup**：构建和打包

### 7.2 可选工具
- **Prettier**：代码格式化（如果团队使用）
- **Husky**：Git 钩子（如果团队使用）
- **Commitlint**：提交消息规范（如果团队使用）

### 7.3 可用命令
```bash
# 运行所有测试
npm test             # vitest run（30秒超时，全局 enable）

# 运行 TypeScript 类型检查
npm run typecheck    # tsc --noEmit

# 构建
npm run build        # tsup → dist/（ESM + CJS + .d.ts）

# 运行 ESLint
npm run lint         # eslint src/
```

## 8. 错误处理指南

### 8.1 错误类型
使用项目定义的错误类型（见 `src/errors.ts`）：
- `ConfigurationError` - 配置错误
- `ValidationError` - 验证错误
- `ExecutionError` - 执行错误
- `TimeoutError` - 超时错误

### 8.2 错误信息
错误信息必须包含：
- 错误类型
- 错误描述
- 相关上下文（如参数值、状态等）
- 建议的解决方案

### 8.3 错误恢复
- 提供重试机制（如果适用）
- 提供降级策略（如果适用）
- 提供回滚方案（如果适用）

## 9. 性能考虑

### 9.1 并发控制
- 使用 `ConcurrencyLimiter` 控制并发
- 避免创建过多并发任务
- 监控内存使用情况

### 9.2 缓存策略
- 使用项目的缓存机制
- 避免重复计算
- 监控缓存命中率

### 9.3 内存管理
- 避免内存泄漏
- 及时释放不需要的资源
- 监控内存使用情况

## 10. 安全考虑

### 10.1 输入验证
- 验证所有外部输入
- 防止注入攻击
- 验证数据类型和范围

### 10.2 敏感信息
- 不硬编码敏感信息
- 使用环境变量
- 遵循最小权限原则

### 10.3 依赖安全
- 定期更新依赖
- 审计依赖安全性
- 避免使用已知有漏洞的版本
