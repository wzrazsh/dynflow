# Dynamic Workflow Engine - 架构决策记录

## ADR-001: 零运行时依赖

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

项目需要决定是否引入外部运行时依赖。

### 决策

**采用零运行时依赖策略**，只使用 Node.js 18+ 内置模块：
- `fetch` - HTTP 请求
- `fs/promises` - 文件系统操作
- `crypto.randomUUID()` - UUID 生成

### 理由

1. **减少供应链攻击风险**
2. **简化部署和维护**
3. **提高可移植性**
4. **更容易审计和安全性审查**

### 影响

- ✅ 更小的包体积
- ✅ 更快的安装时间
- ✅ 更高的安全性
- ⚠️ 需要自己实现一些功能（如重试、超时）

### 后续

- 继续监控 Node.js 内置 API 的变化
- 考虑将常用工具函数提取为内部模块

## ADR-002: 阶段顺序执行，任务并行执行

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定工作流的执行模型：阶段和任务的执行顺序。

### 决策

**阶段顺序执行，任务并行执行**：
- 阶段按顺序执行：阶段1完成后开始阶段2
- 阶段内任务并行执行，受并发限制

### 理由

1. **简单性**：顺序阶段易于理解和调试
2. **可预测性**：阶段间数据依赖明确
3. **性能**：任务并行执行提高吞吐量
4. **错误隔离**：单个任务失败不影响其他任务

### 影响

- ✅ 简化的执行模型
- ✅ 可预测的数据流
- ✅ 良好的错误隔离
- ⚠️ 阶段间无法并行（DAG场景不支持）

### 后续

- 监控用户反馈，考虑支持更复杂的执行模型
- 评估 DAG 执行的复杂性和收益

## ADR-003: 两种API风格

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要提供简单易用的 API，同时保持灵活性。

### 决策

**提供两种API风格**：
1. **Config API**：`Workflow.from({ name, phases, llm })` - 简单直接
2. **Builder API**：`Workflow.define('x').phase('research').task(...)` - 灵活可链式调用

### 理由

1. **简单性**：Config API 覆盖 80% 的使用场景
2. **灵活性**：Builder API 支持复杂工作流定义
3. **渐进式复杂度**：从简单开始，按需使用复杂功能
4. **类型安全**：两种 API 都提供完整的 TypeScript 类型

### 影响

- ✅ 覆盖不同复杂度的使用场景
- ✅ 良好的开发者体验
- ⚠️ 需要维护两套 API

### 后续

- 收集用户反馈，优化 API 设计
- 考虑统一 API 风格（如果用户反馈支持）

## ADR-004: 缓存基于标识而非内容

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定缓存策略：基于内容哈希还是基于标识。

### 决策

**缓存基于标识**：`phaseName:agentId`，而非内容哈希。

### 理由

1. **简单性**：实现简单，易于理解
2. **可预测性**：缓存行为明确
3. **会话恢复**：支持跨会话复用
4. **性能**：避免内容哈希计算开销

### 影响

- ✅ 实现简单
- ✅ 行为可预测
- ✅ 支持会话恢复
- ⚠️ 相同内容可能重复计算（如果ID不同）

### 后续

- 监控缓存命中率
- 考虑添加内容哈希选项（如果需要）

## ADR-005: 错误不传播，状态标记

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定错误处理策略：抛出异常还是状态标记。

### 决策

**错误不传播，状态标记**：
- 任务错误被捕获，存储在 `AgentResult.error` 中
- 任务状态标记为 `'error'`
- 其他任务继续执行

### 理由

1. **容错性**：单个任务失败不中断整个工作流
2. **可观察性**：错误信息完整保留
3. **灵活性**：用户可以决定如何处理错误
4. **调试友好**：错误信息清晰，易于调试

### 影响

- ✅ 良好的容错性
- ✅ 完整的错误信息
- ✅ 灵活的错误处理
- ⚠️ 需要用户主动检查错误状态

### 后续

- 提供错误处理最佳实践
- 考虑添加错误事件通知

## ADR-006: 会话恢复粒度为阶段级

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定会话恢复的粒度：阶段级还是任务级。

### 决策

**会话恢复粒度为阶段级**：
- 已完成的阶段会被跳过
- 中断的阶段会重新执行所有任务

### 理由

1. **简单性**：实现简单，易于理解
2. **可靠性**：阶段级状态更可靠
3. **性能**：避免任务级状态管理的开销
4. **可预测性**：恢复行为明确

### 影响

- ✅ 实现简单
- ✅ 状态管理可靠
- ✅ 恢复行为可预测
- ⚠️ 中断的阶段会重新执行所有任务

### 后续

- 监控恢复成功率
- 考虑添加任务级恢复选项（如果需要）

### 影响

- ✅ 良好的性能

## ADR-008: TypeScript 优先，ESM 模块

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定模块系统和类型安全策略。

### 决策

**TypeScript 优先，ESM 模块**：
- 所有源代码使用 TypeScript
- 输出 ESM 和 CJS 双格式
- 所有导入使用 `.js` 扩展名

### 理由

1. **类型安全**：完整的 TypeScript 类型
2. **现代性**：ESM 是现代 Node.js 的标准
3. **兼容性**：同时支持 ESM 和 CJS
4. **开发体验**：良好的 IDE 支持

### 影响

- ✅ 完整的类型安全
- ✅ 现代模块系统
- ✅ 良好的开发体验
- ⚠️ 需要处理模块解析问题

### 后续

- 监控 Node.js 模块系统变化
- 考虑迁移到纯 ESM（如果社区支持）

## ADR-010: 文档作为上下文控制系统

**状态：** 已接受  
**日期：** 2026年5月  
**决策者：** 开发团队

### 背景

需要决定文档策略：传统文档还是 Vibe Coding 文档。

### 决策

**文档作为上下文控制系统**：
- 文档不是交付物，而是 AI 协作时的上下文控制系统
- 简短、有效、可更新
- 可被 AI 直接读取和工具解析

### 理由

1. **AI 协作**：支持 AI 生成代码 → 自动验证 → 反馈迭代
2. **上下文控制**：防止 AI 误解或遗忘项目方向
3. **轻量性**：避免传统文档的臃肿
4. **可维护性**：文档与代码同步更新

### 影响

- ✅ 良好的 AI 协作体验
- ✅ 轻量级文档体系
- ✅ 可维护的文档
- ⚠️ 需要团队适应新的文档习惯

### 后续

- 收集用户反馈，优化文档结构
- 考虑添加自动化文档生成工具
